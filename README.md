# RLM Benchmark

[![Repo](https://img.shields.io/badge/repo-aleksandarlabs%2Frlm--benchmark-blue)](https://github.com/aleksandarlabs/rlm-benchmark)

A reusable harness to compare three ways of giving a language model access to a codebase that exceeds its context window:

1. **Vanilla** — dump as much code as fits into the prompt and ask.
2. **Terminal agent** — give the model `bash`/`read` tools and let it explore.
3. **RLM (Recursive Language Model)** — load the corpus as a REPL variable and let the model navigate it recursively.

This repo is model-agnostic and corpus-agnostic: drop any codebase into `./corpus`, generate questions with the included agent skill, and run the benchmark.

## Main finding

> For code-understanding tasks, **a terminal agent matched or beat RLM in quality while costing 3-7× less.** RLM's real advantage is in a narrow regime: semantic reasoning over millions of tokens that neither fit in the window nor can be resolved with shell commands. For most engineering work, the terminal already gives you what RLM promises.

Read the full analysis on the [AleksandarLabs blog](https://aleksandarlabs.com).

## Repository structure

```
.
├── corpus/                         # put your target codebase here
├── questions/                      # benchmark questions + ground truth (generated by you)
├── results/                        # raw run outputs (gitignored)
├── skills/
│   └── question-builder/
│       └── SKILL.md                # agent skill to generate questions
├── benchmark/
│   ├── config.py                   # models, providers, conditions
│   ├── corpus.py                   # read and cache the codebase
│   ├── scoring.py                  # needle matching + LLM judge
│   ├── scoring_complexity.py       # F1 set-based scoring
│   ├── vanilla.py                  # vanilla arm
│   ├── rlm.py                      # RLM arm
│   ├── pi_agent.py                 # terminal-agent arm (via Pi CLI)
│   ├── run_bench.py                # Experiment 1: needles + synthesis
│   ├── run_complexity.py           # Experiment 2: crossover by task complexity
│   ├── compare.py                  # compare Exp1 conditions
│   └── rejudge.py                  # re-score synthesis answers
├── examples/                       # example question sets + original run outputs (repomix)
├── pyproject.toml
├── requirements.txt
└── README.md                       # you are here
```

## Requirements

- Python 3.11+
- API keys for the providers you configure (default: Groq + OpenRouter)
- The [`pi`](https://github.com/aleksandar-labs/pi) CLI installed separately for the terminal-agent arm

## Quick start

### 1. Install

```bash
python -m venv .venv
source .venv/bin/activate  # Windows: .venv\Scripts\activate
pip install -r requirements.txt
```

### 2. Configure secrets

Create `.env` at the repo root:

```bash
GROQ_API_KEY=gsk_...
OPENROUTER_API_KEY=sk-or-...
```

### 3. Add a codebase

```bash
# Example: clone repomix as the benchmark corpus
git clone https://github.com/yamadashy/repomix.git corpus/repomix
ln -s corpus/repomix corpus   # or simply move/copy it to ./corpus
```

The benchmark scripts default to `./corpus`. You can override with `--corpus-dir`.

### 4. Generate questions

This is the hardest part of any codebase benchmark. Use the agent skill in `skills/question-builder/SKILL.md`:

1. Load `SKILL.md` into your agent (Hermes, Claude Code, Codex, etc.).
2. Point the agent at `./corpus`.
3. The agent will write `questions/questions.json` and `questions/questions-complexity.json`.

Alternatively, copy the example files for the repomix corpus:

```bash
cp examples/questions-repomix-exp1.json questions/questions.json
cp examples/questions-repomix-exp2.json questions/questions-complexity.json
```

### 5. Measure the corpus

```bash
python benchmark/corpus.py
```

### 6. Run Experiment 1

```bash
python benchmark/run_bench.py --condition vanilla-small --set needles
python benchmark/run_bench.py --condition rlm-strong-subs --set all
python benchmark/compare.py
```

### 7. Run Experiment 2

```bash
# Cheap smoke test: one task per tier, all three arms
python benchmark/run_complexity.py --limit-per-tier 1

# Full matrix
python benchmark/run_complexity.py
```

## Configuring models and conditions

All model IDs, prices, and conditions live in `benchmark/config.py` and can be overridden via environment variables:

| Variable | Default | Meaning |
|---|---|---|
| `GROQ_API_KEY` | — | API key for Groq |
| `OPENROUTER_API_KEY` | — | API key for OpenRouter |
| `GROQ_SMALL_MODEL` | `qwen/qwen3.6-27b` | Small model ID on Groq |
| `RLM_STRONG_MODEL` | `moonshotai/kimi-k2.6` | Strong model ID on OpenRouter |
| `RLM_STRONG_PROVIDER` | `moonshotai/int4` | Pin OpenRouter to one provider |
| `BENCH_JUDGE_MODEL_ID` | `openai/gpt-5-mini` | LLM-as-judge for synthesis |

Conditions (Experiment 1):

```python
"vanilla-small"   # small model, prompt truncated to its window
"vanilla-strong"  # strong long-context model
"rlm-strong-subs" # strong root + cheap recursive subs
"rlm-all-small"   # cheap root + cheap recursive subs
```

## How scoring works

- **Needles:** exact-match against an `accept` list.
- **Synthesis:** LLM-as-judge using the rubric in the question.
- **Complexity tasks:** F1 over sets (precision/recall of listed items). Quadratic tasks require a mechanical ground-truth script.

## Reproducing the published article

The article used `repomix` (~1.13M tokens, 1037 files). To reproduce:

```bash
git clone https://github.com/aleksandarlabs/rlm-benchmark.git
cd rlm-benchmark
git clone https://github.com/yamadashy/repomix.git corpus
cp examples/questions-repomix-exp1.json questions/questions.json
cp examples/questions-repomix-exp2.json questions/questions-complexity.json
python benchmark/corpus.py
python benchmark/run_complexity.py
```

## Adding your own codebase

1. Copy or clone it into `./corpus`.
2. Run the question-builder skill against `./corpus`.
3. Adjust `benchmark/config.py` if you want different models.
4. Run the experiments.

The harness does not depend on the original article's corpus; every path is configurable.

## Results

Raw JSON results are written to `./results/`. They include per-question responses, token usage, cost, elapsed time, and F1 scores. This directory is gitignored; keep it locally or publish a curated subset.
